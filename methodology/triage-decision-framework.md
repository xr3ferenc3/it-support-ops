# Triage Decision Framework

## Purpose

This document defines how to assess, classify, and prioritise a support request from the moment
it arrives - before any diagnostic work begins.

Triage is the discipline of making correct decisions quickly under incomplete information.
Done well, it directs effort to where it matters most, ensures critical issues are never
delayed by minor ones, and produces a consistent first response regardless of which technician
handles the ticket.

Triage does not mean resolving the problem. It means correctly understanding what kind of problem
it is, how urgent it is, who is affected, and what happens next.

---

## When to Use This Framework

Apply this framework to every incoming support request:

- Phone calls from users reporting faults
- Tickets submitted through a help desk portal
- Walk-up requests at the service desk
- Email requests forwarded to the IT team
- Alerts generated by monitoring systems
- Requests passed from another technician

The framework takes under two minutes to apply. There is no ticket too small to triage correctly.

---

## Triage is Not Diagnosis

This is the most important distinction in IT support operations.

| Triage | Diagnosis |
|---|---|
| What kind of problem is this? | What specifically is causing this problem? |
| How urgent is it? | What is the root cause? |
| How many people are affected? | What needs to change to fix it? |
| What category does it fall into? | How do we verify it is fixed? |
| Who should handle it? | How do we prevent it recurring? |
| What is the priority? | |

Triage happens first. Diagnosis follows. Mixing them produces misclassified tickets, wrong
priorities, and wasted effort on low-impact issues while critical faults wait.

## Step 1 - Capture the Initial Report

Before classifying anything, record the facts as reported. Do not interpret. Do not filter.
Write what the user said and what you observed.

**Required intake fields:**

| Field | What to Record |
|---|---|
| Reporter name | Full name and department |
| Contact method | Phone / portal / walk-up / email |
| Time reported | Date and time the request was received |
| Device identifier | Hostname, asset tag, or physical location |
| Operating system | Windows / Linux / macOS and version if known |
| Problem as reported | User's exact words - do not paraphrase |
| Error message | Exact text if visible - screenshot if possible |
| Time fault started | When the user first noticed the problem |
| Recent changes | Anything the user changed or that changed around them |
| Actions already taken | What the user has already tried |

**Intake questions - use these in the order listed:**

1. "Can you describe exactly what you're seeing or what you can't do?"
2. "When did this start - do you know the approximate time?"
3. "Did anything change before this started? Updates, new software, moved your desk?"
4. "Is anyone else on your team having the same issue?"
5. "Are you seeing any error messages? Can you read them to me exactly, or take a screenshot?"
6. "What have you already tried to fix it?"

Do not ask leading questions. "Is it the network?" is a leading question. "What can't you do?"
is a neutral question. Users will confirm leading questions even when they are wrong.

---

## Step 2 - Classify the Issue Type

Match the reported problem to one of the following categories. If the problem spans multiple
categories, select the primary category - the one that is causing the user's reported impact.

| Category | Description | Common Examples |
|---|---|---|
| **Network connectivity** | User cannot reach network resources | No internet, cannot access shared drives, VPN failure |
| **Network performance** | Network is reachable but slow or intermittent | Slow file transfers, video calls dropping, latency complaints |
| **Authentication / access** | User cannot log in or access a resource | Password failure, account locked, permissions error |
| **Hardware fault** | Physical component not functioning | Monitor blank, keyboard unresponsive, device not powering on |
| **Software / application** | Application not launching, crashing, or producing errors | Application crashes, licence error, update failure |
| **Operating system** | OS-level fault affecting system function | Boot failure, BSOD, profile corruption, update loop |
| **Printing** | Cannot print or printer producing incorrect output | Printer offline, print queue stuck, driver error |
| **Email / communication** | Email client or messaging platform fault | Cannot send/receive, client not launching, calendar sync failure |
| **Storage / file access** | Cannot access, save, or find files | Mapped drive missing, file locked, disk full warning |
| **Performance** | System running slowly or unresponsive | High CPU or memory, slow boot, application lag |
| **Security event** | Suspected malware, phishing, or unauthorised access | User clicked suspicious link, unexpected account activity |
| **Request / provisioning** | Not a fault - a request for something new | New account, software install, hardware request |

**If you cannot classify the issue within 60 seconds of reading the report:**
The intake information is insufficient. Return to Step 1 and gather more before proceeding.

---

## Step 3 - Assess the Scope

Scope determines whether this is an isolated fault or a broader incident.

**Ask:**

- Is this affecting one user, one device, one team, one floor, or the whole organisation?
- Is a shared service involved - a server, a printer, a switch, an internet connection?
- Are multiple tickets arriving with the same or similar symptoms?

**Scope levels:**

| Scope Level | Definition | Example |
|---|---|---|
| **Isolated** | One user or one device | Single workstation cannot print |
| **Departmental** | One team or one physical area | Sales team cannot access shared drive |
| **Site-wide** | Entire building or location affected | No internet access at branch office |
| **Organisation-wide** | Multiple sites or all users affected | Email server down, VPN service unavailable |

**Scope escalation rule:**

If the scope is departmental or wider, treat this as a potential incident regardless of the
reported symptom. Refer to [`incident-classification-guide.md`](../incidents/incident-classification-guide.md)
immediately.

Multiple isolated tickets with the same symptom arriving within a short window are a scope
escalation signal. Check the ticket queue before classifying a fault as isolated.

---

## Step 4 - Assign Priority

Priority is determined by two factors: **impact** and **urgency**.

**Impact** - how many people or systems are affected, and how severely.

**Urgency** - how quickly the situation will worsen or how time-critical the user's work is.

### Priority Matrix

| | **High Urgency** | **Low Urgency** |
|---|---|---|
| **High Impact** | P1 - Critical | P2 - High |
| **Low Impact** | P3 - Medium | P4 - Low |

### Priority Definitions

**P1 - Critical**

- A business-critical service is unavailable for multiple users or the entire organisation
- Revenue, safety, or regulatory compliance is directly at risk
- No workaround is available

Examples: Email server down, network switch failure affecting the whole office, finance system
unavailable during month-end, security breach in progress.

Target response: Immediate. Begin work within 15 minutes of classification.
Target resolution: 4 hours or less, or escalate to senior engineering.

---

**P2 - High**

- A significant fault affecting one or more users with no acceptable workaround
- A critical user (executive, on-call engineer, essential business function) is fully impacted
- A service is degraded but not completely unavailable for a wider group

Examples: Senior manager cannot access email, VPN failing for a remote team, shared printer
offline for a department with a printing deadline.

Target response: Within 1 hour.
Target resolution: 8 hours or next business day if fault is discovered late in the day.

---

**P3 - Medium**

- A fault affecting one user where a workaround exists
- A non-critical application is unavailable
- Performance is degraded but core work can continue

Examples: User's second monitor not working, application behaving unexpectedly but usable,
slow network speed on one workstation.

Target response: Within 4 hours.
Target resolution: 3 business days.

---

**P4 - Low**

- A minor inconvenience with no impact on productivity
- A cosmetic issue or user preference change
- A request rather than a fault

Examples: User wants a different default application, a keyboard shortcut is not working,
a background is incorrect on a shared machine.

Target response: Within 1 business day.
Target resolution: 5 business days or next scheduled maintenance window.

---

## Step 5 - Determine Routing

Based on issue type, scope, and priority, determine who handles this ticket.

| Condition | Routing Decision |
|---|---|
| P1 or P2 with site-wide or organisation-wide scope | Raise incident - notify senior engineering immediately |
| P1 or P2 with isolated scope | Assign to available senior technician - do not queue |
| P3 with confirmed network fault | Assign to network-capable technician |
| P3 or P4 with hardware or software fault | Assign to desktop support queue |
| Security event at any priority | Escalate to security or senior engineering regardless of scope |
| Request (no fault) | Route to provisioning or service request queue |
| Fault outside IT scope | Route to relevant team and notify user |

**Routing is not guessing.** If the correct route is unclear, assign to the triage technician's
own queue and seek guidance from a senior technician before routing incorrectly.

---

## Step 6 - Set User Expectation

Before closing the intake interaction with the user, communicate three things:

1. **Acknowledgement** - Confirm you have recorded their issue
2. **Priority** - Tell them what level of urgency has been assigned
3. **Next step** - Tell them what happens next and when they can expect contact

**Example - P3 ticket:**

> "I've logged your ticket for the application issue. This has been classified as medium priority.
> A technician will be in contact within four hours. Your ticket reference is [number].
> If the situation changes - for example, if more people are affected or you have a deadline
> that makes this more urgent - please call us back and we'll reassess."

**Do not:**

- Promise a resolution time you cannot guarantee
- Tell the user what the cause is before diagnosis is complete
- Tell the user the fix before you have confirmed it works
- Close the intake without giving the user a ticket reference number

## Triage Decision Tree

```
Support request received
          │
          ▼
   Step 1: Capture intake
   (name, device, OS, exact symptoms, time, changes, actions taken)
          │
          ▼
   Step 2: Classify issue type
   (network / auth / hardware / software / OS / printing /
    performance / security / request)
          │
          ├── Cannot classify? ──► Return to Step 1, gather more information
          │
          ▼
   Step 3: Assess scope
   (isolated / departmental / site-wide / organisation-wide)
          │
          ├── Departmental or wider? ──► Check incident-classification-guide.md
          │
          ├── Multiple similar tickets arriving? ──► Treat as potential incident
          │
          ▼
   Step 4: Assign priority
   (impact × urgency matrix → P1 / P2 / P3 / P4)
          │
          ├── Security event? ──► Escalate immediately regardless of priority
          │
          ▼
   Step 5: Route the ticket
   (senior tech / desktop queue / network tech / incident / provisioning)
          │
          ▼
   Step 6: Set user expectation
   (acknowledge / priority / next step / ticket reference)
          │
          ▼
   Triage complete - ticket enters appropriate queue
```

## Triage Quality Checks

Before completing triage, verify all of the following:

- [ ] Problem statement is written in observable, factual terms - not assumptions
- [ ] Issue type is assigned to exactly one primary category
- [ ] Scope is confirmed - not assumed based on the user's report alone
- [ ] Priority reflects both impact and urgency - not just the user's perception of urgency
- [ ] Ticket has been routed to the correct queue or technician
- [ ] User has been given acknowledgement, priority, and next step
- [ ] Ticket reference number has been provided to the user
- [ ] All intake fields are complete - no blank fields in the required set

---

## Common Triage Errors

| Error | Effect | Prevention |
|---|---|---|
| Accepting the user's diagnosis | Misclassification, wrong queue | Record symptoms only - not conclusions |
| Skipping scope assessment | Isolated ticket masks a wider incident | Always check for related tickets |
| Assigning priority based on user tone | Urgent-sounding P4 displaces genuine P2 | Use the impact × urgency matrix, not emotion |
| Routing without verifying classification | Ticket reaches wrong team | Confirm type and scope before routing |
| Failing to set user expectation | User calls back repeatedly for status | Always close intake with the three-point update |
| Treating a security event as a standard fault | Delayed response to a breach | Security events are always escalated immediately |

---

## Security Considerations

- Never discuss the scope of a security event with the reporting user beyond confirming it
  has been escalated - scope information can alert an internal threat actor
- Treat any report of unexpected account activity, suspicious emails, or ransomware behaviour
  as a security incident at P1 regardless of apparent scope
- Do not attempt to diagnose or remediate a suspected security event at triage level -
  escalate and contain

---

## Escalation Criteria

Escalate during triage (before diagnosis begins) if any of the following apply:

- Scope is site-wide or organisation-wide
- Priority is P1
- Issue type is a security event
- The same fault is being reported by multiple users simultaneously
- The fault involves a system you do not have access to diagnose

See [`escalation-matrix.md`](escalation-matrix.md) for full escalation procedures and
required information by tier.
